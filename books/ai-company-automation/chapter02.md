# 第2章: 16エージェント設計術 ── .claude/agents/の構造とMDの書き方

## エージェントとは何か（Claude Code の文脈で）

Claude Code には「エージェント」という概念があります。

一言で言うと、**特定の役割に特化した、状態を持つ Claude の分身**です。

通常の Claude との違いは3点です。

1. **専用の人格と知識を持つ**：MDファイルに書いた指示が、そのエージェントの「頭の中」になります
2. **セッションをまたいで記憶を引き継げる**：HANDOFF.md という引き継ぎファイルを毎回読み込むことで、「前回の続き」から始められます
3. **Claude Code が自動で選択する**：どのエージェントを使うかを毎回指定しなくても、`description` フィールドの記述を元に Claude が適切なエージェントを選んでくれます

エージェントを定義するファイルは `.claude/agents/` ディレクトリに置きます。ファイル名が `エージェント名.md` になります。

```bash
.claude/agents/
├── ceo.md          # CEO（戦略統括）
├── yakujihan.md    # 薬自販機営業
├── accea.md        # アクセア印刷サービス営業
├── audit.md        # 監査（デビルズアドボケイト）
├── trainer.md      # エージェント育成部
├── listmaking.md   # リスト部
├── system.md       # システム部
├── marketing.md    # マーケティング
...（全16ファイル）
```

## MDファイルの構造

各エージェントの MD ファイルは3つのパーツで構成されています。

### パート1: フロントマター（YAML）

```markdown
---
name: yakujihan
description: 薬の自動販売機（OTC医薬品IoT販売機・スマートメッズ24）の営業エージェント。薬局・ドラッグストア・ホテル・駅・商業施設向けのリード獲得、提案書作成、メール文作成、テレアポスクリプト、フォロー管理、Gmailログ確認、薬局アプローチ戦略立案など。「薬自販機」「OTC」「ブイシンク」「スマートメッズ」「薬局営業」「ホテル設置」「設置場所」に関する作業はこのエージェントに依頼する。
model: sonnet
---
```

`name` はエージェントの識別子です。`/agent yakujihan` のように直接呼び出す時にも使います。

`description` は **ディスパッチに使われます**。Claude Code がユーザーの入力を受け取ったとき、全エージェントの description を比較して最適なものを選びます。「薬局」「OTC」「スマートメッズ」などのキーワードを入れておかないと、関係ない依頼でも起動されたり、逆に必要な時に選ばれなかったりします。

`model` は使用モデルの指定です。このシステムでは以下の使い分けをしています。

| モデル | 使っているエージェント | 理由 |
|---|---|---|
| `opus` | ceo / trainer / audit | 戦略立案・評価・批判的思考が必要 |
| `sonnet` | yakujihan / listmaking / 多くの実務系 | コスト・速度・品質のバランス |

### パート2: @ファイル参照

フロントマターの直後に `@` で始まる行を書くと、そのエージェントが起動するたびにそのファイルを自動でコンテキストに読み込みます。

```markdown
@/root/ai-sales-company/output/yakujihan/HANDOFF.md
@/root/ai-sales-company/output/yakujihan/lessons_learned.md
@/root/ai-sales-company/memory/company_lessons.md
```

`HANDOFF.md` が「引き継ぎノート」です。前回セッションで何をしたか、次に何をすべきか、商談の進捗がどこまでか、これを毎回自動で読み込むことで、会話の最初から文脈が整った状態になります。

`lessons_learned.md` は過去の失敗事例集です。エージェントが同じミスを繰り返さないよう、オーナーが「NG」と言うたびに教訓が追記されていきます。

**注意点が一つあります**。@include で読み込まれるファイルは肥大化するとセッション開始が重くなります。trainer.md のドキュメントには「150行以内を目安に保つ」と明記してあります。HANDOFF.md が長くなりすぎたら `HANDOFF_archive.md` に旧記録を退避し、本体は要点だけにします。

### パート3: 本文（Markdown）

本文がエージェントの「職務記述書」になります。実際の yakujihan.md の構造を見ると、以下のセクションで構成されています。

```
## アイデンティティ     ← 誰であるか、何に執念を持つか
## ⚠️ 必ず守る禁止事項  ← 過去の失敗から蓄積したNG集
## 商品スペック         ← 暗記必須の数値（毎回調べずに即答できるように）
## 現在の営業戦略       ← 月次で更新する方針
## 参照資料             ← Notion・Google Drive・GmailのURL
## 担当業務             ← 具体的なタスクの手順
## 成果物の品質基準     ← アウトプットの最低ライン
## セッション管理ルール ← 開始時・終了時の必須手順
## 🧩 活用スキル        ← どのスキルをどんな時に使うか
## 🧰 強化ツール        ← MCPツールの使い方
```

## 16エージェントの役割分担設計

なぜ 16 という数なのかは第1章で触れました。ここでは「どう分けるか」の設計思想を説明します。

### 専門化の原則

一つのエージェントに複数の商材を担当させると、コンテキスト汚染が起きます。薬自販機の営業スタイルと印刷サービスの営業では、ターゲット・訴求軸・法的知識がまったく異なります。

そのため、商材ごとに営業エージェントを分けています。

```
yakujihan   → 薬の自動販売機（OTC医薬品IoT販売機）
accea       → アクセア永田町店（印刷サービス）
shinanoagri → シナノアグリ（高級りんご贈答品）
```

### 機能別分離の原則

「何でも一人でやる」エージェントは必ず何かを取りこぼします。リスト作成・提案書作成・システム構築・品質チェックは、専門エージェントとして分離しています。

```
listmaking  → リスト作成に特化（スクレイピング・スコアリング）
system      → データ基盤の構築・保守・分析（GAS・SQLite・CRM）
audit       → 全成果物の批判的レビュー（デビルズアドボケイト）
trainer     → 全エージェントMDの評価・改善（メタエージェント）
```

yakujihan.md には「リスト作成はリスト部に依頼する」「システム構築はシステム部に依頼する」と明示されています。エージェント間の役割境界線をMDに書いておかないと、どのエージェントも全部やろうとして混乱します。

### 情報収集の常設化

週次・月次で自動収集したい情報は、専任エージェントとして常設しています。

```
aiinfo      → 生成AI週次レポート（毎週月曜）
globalinfo  → 海外ビジネス週次レポート（毎週月曜）
news        → 業界ニュース収集（毎週）
bvsync_intel → 薬自販機競合・市場調査
kabutan     → 株探銘柄分析
```

スケジューラから起動するだけで、朝にはレポートが揃っている状態になります。

## エージェント間の情報共有ルール（HANDOFFファイル）

16のエージェントがバラバラに動くと情報が分散します。情報共有の基盤が `HANDOFF.md` です。

### 構造

各エージェントは自分専用の HANDOFF.md を持ちます。

```
output/
├── yakujihan/
│   └── HANDOFF.md   ← yakujihan専用
├── audit/
│   └── HANDOFF.md   ← audit専用
├── trainer/
│   └── HANDOFF.md   ← trainer専用
└── ...
```

会社全体の引き継ぎは `/root/ai-sales-company/HANDOFF.md`（ルート直下）で、CEO エージェントだけが読み書きします。

### 引き継ぎのフォーマット

HANDOFF.md に何を書くかは「次のセッションの最初に知っておくべきこと」に絞ります。yakujihan.md のセッション管理ルールにはこう書かれています。

```markdown
記録する内容：商談進捗・作成した資料・送ったメール・
次回アクション・反応があった施策・Gmailで把握した新情報・
bvsync_intelへの引き継ぎ依頼

⚠️ HANDOFFを肥大化させない：常に「直近の要点・未完了タスク・
次回アクション」に絞る。完了して参照不要になった古い記録は
同フォルダのHANDOFF_archive.mdに移し、本体は概ね150行以内を目安に保つ
```

### エージェント間引き継ぎ

あるエージェントが別のエージェントへ仕事を渡す時も HANDOFF.md を使います。yakujihan.md には次の手順が書かれています。

```markdown
無応答・失注案件は bvsync_intel に引き継ぐ
引き継ぎ手順：
1. HANDOFF.md の「bvsync_intel 引き継ぎ依頼」セクションに記録する
   - 企業名・状況（無応答〇週間 / 失注理由）
   - 監視してほしいこと（例：競合動向、規制進捗、施設の改修情報）
2. Slackで @市場調査AI にメンション
→ bvsync_intel がタイミングを検知したら yakujihan に共有する
```

## 実際のエージェントMDコード例

ここでは実際のファイルから抜粋して、「良い MD の書き方」のポイントを解説します。

### アイデンティティの書き方

抽象的な形容詞を並べても意味がありません。「何に執念を持つか」を一行で示すのが有効です。

yakujihan.md の例:

```markdown
## アイデンティティ
あなたはOTC医薬品自販機（スマートメッズ24）の凄腕営業AIだ。
薬局・ホテル・交通施設・商業施設など業種を問わず、相手の立場で考える。
- 薬局相手には：制度加算・コスト削減・24時間対応の観点で話す
- ホテル・施設相手には：インバウンド対応・スタッフ不要・収益多様化の観点で話す
```

ceo.md の例:

```markdown
## アイデンティティ
あなたは売上最大化に執念を持つ敏腕経営者AIだ。
ハーバードMBA・マッキンゼー出身の戦略思考と、
叩き上げ営業マンの現場感覚を併せ持つ。
「できない理由」より「どうすればできるか」を常に考える。
```

### 禁止事項の書き方

禁止事項には**理由を添える**のが重要です。理由がない禁止は形骸化します。

```markdown
## ⚠️ 必ず守る禁止事項

### 重大NG（過去に複数回繰り返した禁止事項）
- **アパホテルと羽田空港は設置候補リストに絶対に記載しない**
  → 明示的に指示されているにもかかわらず複数回混入させた重大NG
```

「複数回繰り返した」という記録が重要です。エージェントは同じミスを繰り返します。1回目の失敗で MD に書かなければ、2回目・3回目が起きます。

### 数値は MD に直接埋め込む

「スペックを調べて」とエージェントに指示するより、MD に数値を書いておく方が速く・正確になります。

```markdown
## 商品スペック（暗記必須・資料なしでもこの数値で即答する）

| 項目 | 数値・内容 |
|---|---|
| 本体価格 | 348万円（税別） |
| 月次コスト（補助金＋リース活用） | 月3.8万円台〜 |
| 日次損益分岐ライン | 約2.3万円/日（月70万円達成目安） |
| 保証期間 | 7年保守保証 |
| 法改正施行予定日 | 2027年5月26日（改正薬機法・遠隔販売関連） |
```

### セッション開始・終了の手順を明示する

「必要な時に考える」ではなく、定型手順として MD に書いておきます。

```markdown
## セッション管理ルール

### 開始時（必須・毎回この順番で）
1. HANDOFF.md 確認（@import で自動読込済み）：前回の商談進捗・
   未完了タスク・引き継ぎ事項を把握する
2. Gmailログシート確認：スプレッドシートの「Gmailログ」シートを開き、
   ★★★のメールを最優先でチェック
3. Notion 最新3回分を読む：notion-fetch で直近3回の定例ページを取得

### 終了時（必須）
/root/ai-sales-company/output/yakujihan/HANDOFF.md を更新する
```

## trainer.md ── メタエージェントの設計

16エージェントの中で一つ特殊な役割があります。`trainer`（エージェント育成部）です。

他のエージェントは商材・業務に向き合います。trainer だけは**エージェントそのものを評価・改善することが仕事**です。

```markdown
---
name: trainer
description: エージェント育成部。全AIエージェントのMD（定義ファイル）を継続的に評価・
改善し、各エージェントを進化させるメタエージェント。チャットログ・成果物・git履歴から
仕事の質と速度を分析し、改善diffを提案する。
model: opus
---
```

trainer.md には評価の6軸が定義されています。

```markdown
## 評価フレームワーク（6軸）

1. 成果の質: 手戻り・オーナー修正指示の回数
2. 速度・効率: 無駄なツール呼び出し・迷走がないか
3. 指示遵守: MDに書かれたルールを守れているか
4. コンテキスト効率: MDが長すぎないか
5. 鮮度: 古い日付・終わった案件が残っていないか
6. モデル適合: 現行Claudeの能力に対して手取り足取りすぎないか
```

そして最も重要な方針が「**足す」より「削る」**」です。

```markdown
信条：
- 「足す」より「削る」。MDは長いほど良いのではない。
  冗長な指示はコンテキストを圧迫し性能を下げる
```

エージェントの MD を書いていると、ついルールを追加し続けてしまいます。しかし長くなるほど、エージェントは重要な指示を見落とします。trainer は週次で全 MD をレビューし、削れるものを探し続けます。

## MD設計のまとめ：実際に使えるチェックリスト

実際に新しいエージェントを作る時に確認するポイントをまとめます。

```
□ フロントマター
  □ name: が決まっているか
  □ description: にトリガーキーワードが入っているか
  □ model: が適切か（思考系 opus / 実務系 sonnet）

□ @ファイル参照
  □ HANDOFF.md を @include しているか
  □ lessons_learned.md を @include しているか
  □ @include するファイルが肥大化しないよう管理方針を決めているか

□ 本文
  □ アイデンティティが「何に執念を持つか」で書かれているか
  □ 担当業務と担当外業務の境界が明示されているか
  □ 禁止事項に理由が付いているか
  □ セッション開始・終了の手順が番号付きで書かれているか
  □ 成果物の保存先（絶対パス）が明記されているか
  □ 活用スキルと強化ツールの一覧があるか
```
